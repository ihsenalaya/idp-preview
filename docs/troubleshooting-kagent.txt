-------------------------------------------------------------------------------
CONTEXTE
-------------------------------------------------------------------------------
L'opérateur déclenche une analyse kagent via le protocole A2A (JSON-RPC 2.0)
lorsqu'une suite de tests échoue (contract, regression, e2e). Il appelle
l'agent preview-troubleshooter-agent en HTTP POST sur le port 8080, attend la
réponse, puis poste le résultat en commentaire GitHub sur la PR concernée.

-------------------------------------------------------------------------------
PROBLÈME 1 — "agent returned failed state" (HTTP 400 Azure OpenAI)
-------------------------------------------------------------------------------

Symptôme :
  Le log opérateur affiche en boucle :
    ERROR kagent analysis failed {"error": "agent returned failed state"}
  La réponse A2A contient bien result.status.state = "failed".
  L'agent répond en ~14 secondes (trop rapide pour une analyse complète).

Diagnostic :
  Inspection des logs de l'agent pod (preview-troubleshooter-agent) :
    18:28:45 - POST Azure OpenAI → HTTP 200 OK  (1er appel LLM)
    18:28:55 - POST Azure OpenAI → HTTP 200 OK  (2ème appel LLM)
    18:28:57 - POST Azure OpenAI → HTTP 400 Bad Request  ← ÉCHEC ici
    18:28:57 - runner closed, state = failed

  Le 3ème appel LLM est rejeté par Azure OpenAI avec HTTP 400 :
  cause = context window exceeded (token limit dépassé).

  Enchaînement des appels d'outils par l'agent :
    Tour 1 : LLM décide d'appeler k8s_get_resources + k8s_get_pod_logs
    Tour 2 : Les résultats des logs de jobs (smoke, contract, e2e...)
             sont très verbeux → beaucoup de tokens dans l'historique
    Tour 3 : LLM tente de faire un appel Jaeger, mais le contexte
             cumulé (prompt + tool results) dépasse gpt-4o-mini

Tentatives échouées :
  - Penser que c'était un problème réseau ou NetworkPolicy :
      Test depuis kagent-system → completed
      Test depuis preview-operator-system → completed
      Seul l'opérateur (processus Go) échouait → piste réseau abandonnée
  - Penser à un problème de concurrence (debug pod vs opérateur) :
      Les deux calls étaient bien séquentiels, pas un race condition
  - Penser à une session A2A corrompue :
      Chaque call utilise uuid.New() → sessions toujours fraîches

Solution appliquée (v1.0.13) :
  Modifier buildAnalysisPrompt() dans internal/controller/kagent.go pour
  inclure directement les sorties de tests (lignes PASS/FAIL) dans le
  prompt initial. L'agent reçoit les échecs sans avoir besoin de fetch
  des logs k8s → moins de tokens de tool results → le 3ème appel LLM
  reste dans les limites du contexte.

  Avant :
    Prompt minimal → agent fait 3-4 tours de tool calls → overflow

  Après :
    Prompt inclut status.tests.*.output (max 10 lignes par suite)
    → l'agent fait 1-2 tours → pas d'overflow

  Commit : c825598 — preview-operator/internal/controller/kagent.go

Vérification :
  18:50:21 - INFO kagent analysis posted to GitHub {"commentId": 4413413480}
  Le commentaire contient : risk=HIGH, evidence, suggested fix, kubectl commands

PROBLÈME 2 — Idempotence : kagent ne se relance pas automatiquement
-------------------------------------------------------------------------------

Symptôme :
  Après un échec kagent, l'opérateur réessaie toutes les 5 minutes (cooldown).
  Mais si aucun événement k8s ne survient après les tests, le réconcile
  ne se déclenche plus et le retry ne se fait jamais.

Diagnostic :
  Le reconcile loop se termine sans RequeueAfter quand les tests sont finis.
  Le status update de kagent (phase=Failed) triggère un réconcile, mais le
  cooldown (5 min) bloque. Après 5 min, s'il n'y a pas de nouvel événement
  (plus de jobs qui changent d'état), l'opérateur reste silencieux.

Solution de contournement :
  Patcher l'annotation de la CR pour forcer un réconcile :
    kubectl patch preview pr-4 --type=merge \
      -p '{"metadata":{"annotations":{"kagent.preview.io/retry":"<timestamp>"}}}'

  Solution permanente à implémenter : ajouter un RequeueAfter=5min
  dans le reconcile quand kagent.phase=Failed et CommentID=0.
  (pas encore implémenté)

PROBLÈME 3 — preview-troubleshooter-agent ACCEPTED=False (jaeger-mcp-server manquant)
-------------------------------------------------------------------------------

Symptôme :
  kubectl get agent preview-troubleshooter-agent -n kagent-system
  → READY=(vide)  ACCEPTED=False

  kubectl describe agent preview-troubleshooter-agent -n kagent-system
  → Message: "failed to compile agent ... RemoteMCPServer.kagent.dev
    'jaeger-mcp-server' not found"

  L'opérateur reçoit : "A2A call: ... dial tcp: lookup
  preview-troubleshooter-agent.kagent-system.svc.cluster.local: no such host"
  (le service n'est pas créé tant que l'agent n'est pas Accepted)

Cause :
  Le manifest k8s/kagent/jaeger-mcp-server.yaml n'a pas été appliqué.
  L'agent référence le RemoteMCPServer "jaeger-mcp-server" ; sans lui,
  le contrôleur kagent refuse de compiler l'agent.

Solution :
  1. Créer le pull secret GHCR dans kagent-system (l'image est privée) :
       GH_TOKEN=$(cat ~/.config/gh/hosts.yml | grep oauth_token | awk '{print $2}')
       kubectl create secret docker-registry ghcr-pull-secret \
         --namespace kagent-system \
         --docker-server=ghcr.io \
         --docker-username=<GITHUB_USER> \
         --docker-password="$GH_TOKEN"

  2. Appliquer le manifest jaeger-mcp-server :
       kubectl apply -f k8s/kagent/jaeger-mcp-server.yaml
       kubectl rollout status deployment/jaeger-mcp-server -n kagent-system --timeout=120s

  3. Redémarrer l'agent pour qu'il se reconcilie :
       kubectl rollout restart deployment/preview-troubleshooter-agent -n kagent-system

  4. Vérifier :
       kubectl get agent preview-troubleshooter-agent -n kagent-system
       # → READY=True  ACCEPTED=True
       kubectl get svc preview-troubleshooter-agent -n kagent-system
       # → le service doit exister sur le port 8080

Vérification finale :
  Si ACCEPTED=True mais l'analyse échoue encore, voir PROBLÈME 4 ci-dessous.

PROBLÈME 4 — 401 Azure OpenAI : clé corrompue par CRLF (WSL/Windows)
-------------------------------------------------------------------------------

Symptôme :
  Log opérateur :
    ERROR kagent analysis failed {"error": "agent returned failed state"}
  Log agent :
    openai._base_client - INFO - Retrying request to /chat/completions in X seconds
    (2 retries, puis runner closed, state = failed)

  Test direct depuis le pod :
    kubectl exec -n kagent-system deploy/preview-troubleshooter-agent -- python3 -c "
      import os, openai
      client = openai.AzureOpenAI(
        azure_endpoint=os.environ['AZURE_OPENAI_ENDPOINT'],
        api_key=os.environ['AZURE_OPENAI_API_KEY'],
        api_version='2024-10-21'
      )
      r = client.chat.completions.create(model='gpt-4o-mini',
        messages=[{'role':'user','content':'hi'}], max_tokens=5)
      print(r.choices[0].message.content)"
    → AuthenticationError: 401 Access denied due to invalid subscription key

  Inspection du secret :
    kubectl get secret kagent-openai -n kagent-system \
      -o jsonpath='{.data.OPENAI_API_KEY}' | base64 -d | xxd | tail -2
    → 0x0d à la fin de la clé (carriage return \r)

Cause :
  Sur WSL/Windows, az CLI retourne les valeurs avec \r\n en fin de ligne.
  La commande $(az ... --query "key1" -o tsv) capture le \r, qui est
  encodé dans le secret Kubernetes et envoyé tel quel dans le header
  Authorization → Azure OpenAI rejette la clé corrompue.

Solution :
  Toujours utiliser tr -d '\r\n' lors de la capture de valeurs az CLI :

    AOAI_KEY=$(az cognitiveservices account keys list \
      --name "preview-openai" --resource-group "<RG>" \
      --query "key1" -o tsv | tr -d '\r\n')

    # Recréer le secret si déjà créé avec la clé corrompue :
    kubectl delete secret kagent-openai -n kagent-system
    kubectl create secret generic kagent-openai \
      --namespace kagent-system \
      --from-literal=OPENAI_API_KEY="$AOAI_KEY"

    # Redémarrer le controller ET l'agent pour recharger le secret :
    kubectl rollout restart deployment/kagent-controller -n kagent-system
    kubectl rollout restart deployment/preview-troubleshooter-agent -n kagent-system

  Note : l'env var injectée dans le pod est AZURE_OPENAI_API_KEY (pas
  OPENAI_API_KEY) — c'est le kagent-controller qui fait le mapping.

Vérification :
  kubectl get secret kagent-openai -n kagent-system \
    -o jsonpath='{.data.OPENAI_API_KEY}' | base64 -d | xxd | tail -1
  → la dernière ligne ne doit PAS se terminer par 0d (0d = \r)

  Test LLM depuis le pod (voir commande ci-dessus) → doit répondre "OK: ..."

Date de résolution : 2026-05-10
Clusters affectés : prod + test1 (même secret CRLF dans les deux)
